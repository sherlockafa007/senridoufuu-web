# 内容编辑器：所见即所得（就地编辑 + 中文自动出日英）设计

日期：2026-07-21
状态：已与用户确认方向，待实施
关系：本文档取代 `docs/specs/2026-07-14-admin-cms-design.md` 第 6 节（网站内容编辑）的表单方案；
第 3 节（Worker 架构/鉴权/发布链）、第 7 节（图片，仍属下一期）不变，本次不改动。

## 1. 触发背景

用户用现有表单编辑器（`admin/index.html` 的 SECTIONS 数组）改了"关于我们·大事记"，
发现：①表单字段（眉标/标题/简介）与页面实际显示位置的对应关系不直观；②只填了中文标签，
日文标签留空，导致日文页面看起来"没生效"（实际是版本正确，只是没翻译）。

用户明确诉求：编辑界面要和真实页面长得一样、改哪显示哪；以中文为主力语言撰写，
日英应自动派生，不再逐语言手填。

## 2. 现状盘点（写此文档前核实过代码，非推测）

- 全站文字通过 `data-i18n="key"` 属性 + `js/main.js` 的 `applyTranslations()` 渲染，
  已是相当彻底的 i18n 体系。
- 四个页面（`index.html`、`about/index.html`、`about/milestones.html`、`solutions/index.html`）
  加上 `main.js` 注入的导航/页脚，共有 **69 个不同的 `data-i18n` key**。
- 现有表单编辑器（admin SECTIONS 数组）只手工登记了 **57 个**，遗漏 27 处
  （产品卡片、方案页 CTA 按钮、hero 副标题等）——这些字段一直改不了，纯属清单没跟上页面迭代。
- 现有表单还有一个"图片"标签页（`IMAGE_FIELDS`：团队照片、分享图），**但站点渲染逻辑
  从未读取这些覆盖值**——`main.js` 只合并文字（`T[lang]`），从不处理图片 src。
  这是个从未真正工作过的死功能。
- `js/main.js:669` 用 `el.textContent = t[key]` 渲染，`css/main.css` 中没有任何
  `white-space: pre-line` 规则——多行字段（如"使命正文"）里的换行符**在真实页面上会被
  浏览器折叠消失**，是编辑器一直存在但从未被注意到的显示 bug。

## 3. 本次范围

**做**：
- 用"就地编辑"取代表单：后台直接嵌入真实页面，点文字就地改，改完即所见即所得。
- 可编辑字段从页面 `data-i18n` 属性**自动发现**，不再手工维护清单——27 个遗漏字段随之修复。
- 中文撰写 → 一键调用 Qwen 生成日文/英文，存为公用 Worker 接口，供本次编辑器和未来
  Blog 模块共用。
- 修复多行文字换行显示 bug（新编辑器保真的前提）。
- 移除死功能"图片"标签页（避免用户误以为图片能通过它更新）。

**不做（明确排除，YAGNI）**：
- 图片上传/替换——原 spec 第 7 节的下一期任务，工作量独立（需要 `data-img-key` 标记、
  Worker 侧文件提交、客户端压缩），不在本次范围。
- "页面上任意一个字都能点"——只有已接入 `data-i18n` 的 69 处可编辑，与页面装饰性/
  结构性文字（如按钮的固定符号）无关。
- Blog 模块本身——只搭建其将来会复用的翻译组件接口，不实现 Blog 撰写界面。

## 4. 架构

**关键判断：同源，无需 postMessage。**
后台 `/admin/` 与被编辑页面（`/`、`/about/`、`/solutions/`）同属 `www.senridf.com`
同一来源。这意味着后台的 JS 可以**直接访问 iframe 内的 DOM**（`iframe.contentDocument`），
不需要跨域消息传递那一整套复杂度——本地开发（同一静态服务器多端口）与线上（同一域名）
都成立。

```
/admin/（后台）
├─ 页面切换标签：首页 / 关于我们 / 大事记 / 解决方案
└─ <iframe src="/xxx?lang=zh">          ← 加载真实页面，强制中文
     │  （同源，后台 JS 直接操作 iframe.contentDocument）
     ├─ 遍历 iframe 内所有 [data-i18n] 元素 → 加悬浮虚线框
     ├─ 拦截点击：命中 [data-i18n] → 就地 contenteditable；其它 → 阻止默认导航
     │   （编辑模式下 iframe 里的链接/按钮不应该真的跳转）
     └─ 编辑完成 → 记入本地"待发布变更"表（key → 新中文值）

顶部工具条：「已改 N 处」「✨ 一键同步日英」「保存并发布」
     │
     ├─ 一键同步日英 → Worker `/translate`（新增路由，Qwen 批量翻译待发布变更）
     │                  → 结果写回待发布变更表（ja/en 值），不改变 iframe 显示（仍是中文）
     │
     └─ 保存并发布 → 复用既有 Worker `/content` PUT（无需改动）→ 既有发布轮询 UX（无需改动）
```

### 4.1 字段自动发现

后台不再维护 `SECTIONS` 清单。加载某页面 iframe 后：
```js
const editableKeys = [...iframeDoc.querySelectorAll('[data-i18n]')].map(el => el.dataset.i18n);
```
每个 key 的"当前值"直接读元素当前文字（已经是中文渲染结果，因为 iframe 强制 `?lang=zh`）。
对比 `content.json` 里是否已有覆盖，决定显示的是"网站原文"还是"你之前的自定义值"——
对用户来说无感，就是"这里现在显示什么，你就点它改什么"。

### 4.2 编辑交互

- 悬浮：鼠标移到 `[data-i18n]` 元素上，虚线框高亮（后台注入一段轻量 CSS 到 iframe head）。
- 点击：`e.preventDefault()` 阻止任何导航，目标元素设 `contentEditable = true` 并聚焦，
  元素上方浮出小工具条「✓ 完成」「✕ 取消」。
- 粘贴处理：强制纯文本粘贴（阻止带格式内容污染排版）。
- 确认：取 `el.innerText.trim()` 存入待发布变更表，退出编辑态；取消则还原原文字。
- 顶部工具条实时显示"已改 N 处"，未发布前可继续改多处，最终一次性发布（沿用现有
  "攒够了一起提交"的批处理体验，不做成每改一个字就发一次布）。
- 待发布变更表**仅存于当前页面内存**，不做草稿持久化——刷新或关闭页面会丢失未发布的改动，
  与现有表单编辑器的行为一致（无退步）。Blog 模块的草稿持久化需求不同（撰写耗时更长），
  沿用原 spec 第 5 节的 Firestore 草稿方案，与本次无关。

### 4.3 语言强制与站点隔离

`js/main.js` 的语言选择目前读 `localStorage.sdf_lang`（默认 `ja`）。若后台 iframe 也用
这个键，会导致**站长本人正常浏览网站的语言偏好被后台悄悄改成中文**——需避免。

做法：`main.js` 语言初始化时优先读 URL 参数 `?lang=`，其次才是 `localStorage`。
后台加载 iframe 时固定传 `?lang=zh`，两者互不干扰。

### 4.4 中文→日英同步（公用组件）

新增 Worker 路由 `POST /translate`，输入 `{ fields: [{ key, zh }] }`（批量，一次网络请求
翻完本轮所有待发布变更，而非逐字段调用），输出 `{ ja: {...}, en: {...} }`（按 key 对应）。
Prompt 明确要求保持 HTML 无关、纯文本、语气与原站一致，避免管理员改一个字段而模型顺手
"优化"了措辞。

此路由不含发布逻辑，是纯翻译服务——将来 Blog 模块的"写中文自动出日英"直接调用同一路由，
不重新设计一遍。

部分字段翻译失败时（如模型对某一条返回异常），该字段在待发布变更表中保留"未同步"状态，
其余字段正常写入；顶部提示"N 处未生成日英，可重试或留空发布（留空则日英沿用网站原文）"，
不因单条失败阻塞整体发布。

### 4.5 多行文字换行修复

多行字段（对应原 SECTIONS 里 `multiline:true` 的那些：xxx_body、xxx_desc 等）渲染它们的
CSS 选择器补上 `white-space: pre-line`，让 `\n` 正确显示为换行。这是本次改动能"所见即所得"
的必要前提（否则 contenteditable 里换行显示正确，发布后又消失，等于新编辑器说话不算话）。
具体选择器清单在实施计划里核对（读 `css/main.css` 现有类名后落地，避免影响无关元素）。

## 5. 移除内容

`admin/index.html` 里的 `IMAGE_FIELDS`、"图片"标签页、`buildImages()`、`updateImgPreview()`
一并删除——它们从未真正工作过，保留只会继续误导用户。图片编辑作为独立功能在下一期
（原 spec 第 7 节）重新设计并接通渲染逻辑。

## 6. 测试

- 纯逻辑（字段发现的 key 提取、待发布变更表的 diff 计算、翻译请求/响应的数据整形）
  拆成可测模块，进 `node --test` + CI。
- 就地编辑的 DOM 交互（contenteditable、iframe 同源访问、悬浮高亮）**CI 测不了**，
  是浏览器行为——上线前人工过一遍：切四个页面标签、改字段、同步日英、发布、
  刷新真实页面确认生效、多行字段确认换行保留。

## 7. 范围外遗留问题（记录，不在本次做）

- 图片上传/替换（下一期）。
- Blog 撰写界面（复用本次的 `/translate`，界面另设计）。
- 27 个新纳入编辑的字段中，是否有需要保护的（如法务地址）尚未逐一审阅——上线后
  用户可自行核对这些字段的默认值是否合理。
